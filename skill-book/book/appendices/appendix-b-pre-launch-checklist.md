# 附录 B: Skill 发布前检查清单

> 基于 OpenClaw 最佳实践和 Anthropic 官方规范提炼

发布一个 Skill 到团队或社区前，逐项检查以下清单。

---

## 核心质量（6 项）

- [ ] `name` 使用 kebab-case，64 字符以内，不以连字符开头/结尾
- [ ] `description` 精准，包含 What + When + Keywords，第三人称，1024 字符以内
- [ ] 主文件 < 500 行，< 5,000 词
- [ ] 术语统一，无自相矛盾的指令
- [ ] 提供具体示例（至少 1 个输入/输出对照）
- [ ] 所有 references 文件从主文件直接链接（引用深度 = 1）

## 结构完整性（4 项）

- [ ] 包含 "When to Use" 或触发场景描述
- [ ] 包含 Guardrails 或 Anti-Patterns 部分
- [ ] 关键操作包含 VERIFY 验证步骤
- [ ] 长流程有步骤编号或清单

## 安全与可靠性（5 项）

- [ ] 明确列出"不可做"的事项
- [ ] 破坏性操作有明显标记
- [ ] 高权限操作有安全模型声明
- [ ] 无硬编码凭证（API Key, Token, 密码）
- [ ] 默认行为是安全/非破坏性的

## 代码与脚本（4 项）

- [ ] 代码示例可直接运行
- [ ] 脚本遵循规范：JSON→stdout, 状态→stderr, set -e, trap cleanup
- [ ] 关键步骤包含验证命令
- [ ] 路径使用相对路径（从 Skill 根目录计算）

## 测试覆盖（4 项）

- [ ] 至少 3 组测试场景（正常路径 + 边界情况 + 错误处理）
- [ ] 触发测试：5 个 should-trigger + 5 个 should-not-trigger 查询
- [ ] 一致性测试：同一请求运行 3 次，格式和质量稳定
- [ ] 真实工作流端到端验证

---

## 快速检查版（5 分钟速查）

如果时间有限，至少检查以下 5 项：

1. **description 是否精准？** — 能让不了解这个 Skill 的人立刻知道它做什么、何时用
2. **有没有 Guardrails？** — 至少列出 3 条"不可做"的事
3. **有没有示例？** — 至少 1 个完整的输入→输出对照
4. **有没有硬编码？** — 搜索路径、Key、版本号
5. **试运行过吗？** — 在全新的对话中运行一次，不带任何先前上下文
